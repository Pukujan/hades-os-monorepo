# Contract: Repository Catalog

**Version:** v001

## Purpose

Define the content, format, and generation of metadata catalog files under `metadata/` that index the repo's modules, tasks, contracts, APIs, and architecture fitness checks.

## Scope

All files under the `metadata/` directory at the repo root. This directory is generated, not authored.

## Rules

1. `metadata/` contains one JSON file per catalog category.
2. Each file is valid JSON with a top-level object. Arrays may appear as field values.
3. Required catalog files: `repo.json`, `catalog.json`, `modules.json`, `tasks.json`, `contracts.json`, `apis.json`, `architecture-fitness.json`, `dependency-graph.json`.
4. Catalog files are generated by scripts, not manually edited.
5. Each catalog file must include a `"generatedAt"` ISO-8601 timestamp.
6. Each catalog file must include a `"version"` field matching the repo artifact layout version.
7. A `metadata/INDEX.md` must list all catalog files with brief descriptions.

## Required artifacts

- `metadata/repo.json` — repo-level metadata (name, version, root layout)
- `metadata/catalog.json` — cross-catalog index
- `metadata/modules.json` — module inventory
- `metadata/tasks.json` — task inventory
- `metadata/contracts.json` — contract registry (mirrors manifest.json)
- `metadata/apis.json` — API endpoint registry
- `metadata/architecture-fitness.json` — fitness function registry
- `metadata/dependency-graph.json` — module dependency graph
- `metadata/INDEX.md` — catalog index

## Enforcement

Automated: `test:repo-architecture` includes a test for metadata catalog files (expected red until Phase 3).

Manual: verify that generation scripts update the `generatedAt` timestamp.

## Non-goals

- Storing authored content in metadata/ (it is generated only)
- Schema validation beyond JSON parseability
- Live sync with source changes
